# アーキテクチャ

この文書は、現行実装が従う責務分割と、同じ方向で拡張する際の配置基準を定義する。

## ディレクトリ構成

```txt
src/
  app/
    layout.tsx
    page.tsx
    login/
    signup/
    profile/
    admin/upload/

  features/
    auth/
    inventory/
    profile/

  components/
  lib/
  types/
```

`src/app` はルーティング境界である。`page.tsx`、`layout.tsx`、各 URL セグメント配下の画面はここに置く。画面固有の表示ロジックは近接配置できるが、複数画面で再利用される処理は `features`、`components`、`lib` に分ける。

`src/features` はアプリ固有の機能を置く。現在の機能境界は `auth`、`inventory`、`profile` である。認証、在庫表示と更新、プロフィール更新のような振る舞いは、呼び出し元の画面ではなく、その機能ディレクトリが所有する。

`src/components` は機能横断で再利用する UI を置く。単一画面だけに閉じた表示部品は、その画面の近くに置ける。

`src/lib` は外部システムやランタイムとの接続点を置く。現在は Prisma クライアントがここにある。アプリ全体で共有するインフラ境界だけを置き、機能固有の業務判断は持たせない。

`src/types` は複数機能から参照される型契約を置く。現在は在庫状態の契約をここが所有する。特定機能に閉じた型は、その機能ディレクトリ内に置く。

## 配置原則

画面は入出力の境界を担当し、永続化、認証、状態遷移、再検証の判断は owning feature に寄せる。

共通化は再利用回数ではなく責務で判断する。同じ種類の判断を複数箇所で持つときだけ共有モジュールに上げる。

名前は責務を直接表す。抽象的な総称ではなく、何を所有するかがファイル名とディレクトリ名で分かる状態を保つ。

型と値の契約は権威ソースに近い場所で定義する。外部入力の検証と内部表現の変換は、入力境界か owning feature が担当する。

## 拡張時の基準

新しい画面は、まず `src/app` にルート境界を追加し、その画面が依存するアプリ固有ロジックを既存 feature に入れるか、新しい feature を切るかを責務で決める。

新しい feature は、独立した振る舞い、データ取得、更新責務を持つときに追加する。単なる UI の断片や薄いラッパーのために feature を増やさない。

共有型は複数 feature の契約になった時点で `src/types` に置く。それまでは feature 内に留める。
